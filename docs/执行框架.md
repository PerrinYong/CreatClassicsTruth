# 执行框架

## 1. 目的

本文件定义本项目的长期执行方式。

它回答四个问题：

1. 章节如何进入 runloop
2. Agent 如何分工
3. 产物如何落盘和冻结
4. 本地代码只负责什么

---

## 2. 总体执行模式

本项目不通过本地程序批量跑全书，而是通过宿主环境中的顺序式 runloop 执行。

推荐的基本循环：

1. 读取当前章节原文
2. 读取当前冻结状态 `book_state.json`
3. 分派 Extractor / Compressor / State Keeper / Auditor
4. 生成并审计本章产物
5. 写入 checkpoint
6. 更新 `current_chapter_id`
7. 推进到下一章

---

## 3. 推荐 runloop 粒度

### 3.1 书级 runloop

适用于全书首轮跑通。

特点：

- 逐章顺序
- 一章完成后才进入下一章
- 强依赖 `book_state.json`

### 3.2 章级 runloop

适用于单章回修或补审。

特点：

- 只处理一章
- 可单独回放
- 允许围绕本章并行发起辅助审计

### 3.3 后处理 runloop

适用于全书完成后：

- 汇总全书精简文本
- 统一图数据
- 生成展示页
- 生成统计报告

---

## 4. Agent 分工

### 4.1 Coordinator

- 控章节顺序
- 控回修
- 控冻结
- 控最终交付

### 4.2 Extractor

- 从原文抽取真相层
- 必须带 `source_anchor` 和 `source_cues`
- 不做评论性输出

### 4.3 Compressor

- 仅基于 truth 写精简文本
- 不得新增事实

### 4.4 State Keeper

- 更新 `book_state.json`
- 维护 open threads
- 派生时间线和图数据

### 4.5 Auditor

- 检查漏项
- 检查幻觉
- 检查状态冲突
- 决定是否允许冻结

---

## 5. 落盘策略

### 5.1 章节 checkpoint

每章应在 `workspace/checkpoints/<chapter_id>/` 下保留：

- `chapter_truth.json`
- `chapter_retelling.txt`
- `chapter_audit_report.json`
- `chapter_timeline_snapshot.json`
- `chapter_story_flow.json`
- `book_state.json`

### 5.2 工作状态

当前工作状态保存在：

- `workspace/state/book_state.json`

### 5.3 正式输出

全书级输出保存在：

- `workspace/outputs/`

### 5.4 审计报告

阶段性报告保存在：

- `workspace/reports/`

---

## 6. 本地代码边界

### 6.1 可以写的代码

- JSON Schema 校验器
- 目录与文件命名校验器
- 由 truth/state/graph 生成网页展示数据的脚本
- 静态页面生成脚本

### 6.2 不应再写的代码

- LLM API 调用器
- Agent 编排器
- LangGraph 工作流
- 对话状态机
- 自研任务调度器

---

## 7. 长期执行建议

### 7.1 首轮策略

- 单书
- 顺序逐章
- 强审计
- 高频 checkpoint

### 7.2 稳定后策略

- 增加黄金集
- 增加本地 Schema 校验
- 增加图页面输出
- 增加统计和覆盖率报告

### 7.3 明确延后事项

- 跨书统一实体层
- 智慧提炼层
- 大规模并行处理
- 复杂交互式知识图谱后端

---

## 8. 执行入口建议

实际执行时，优先围绕以下对象组织会话：

- 当前章节原文文件
- `workspace/state/book_state.json`
- `framework/contracts/*.schema.json`
- 当前章节 checkpoint 目录

这意味着：

**项目真正的“框架”不是 Python 程序，而是文档、契约、状态文件和目录约定。**

# 名著事实提取项目方案书

## 1. 项目定位

本项目的定位不是“提炼智慧工程”，而是：

**对单本中国古典小说进行忠实的信息与情节提取，生成高保真、可压缩、可计算、可画图的结构化成果。**

项目只做三件核心事：

1. 生成**不丢失关键情节信息的精简版纯文本**
2. 生成**机器可理解的 JSON 真相层**
3. 生成**可直接用于绘制故事发展图、人物关系图、时间线快照图的数据层**

这里的“忠实”，不是逐字保留原文，而是：

> **不丢失所有与故事情节、因果链、人物状态变化、关系变化、阵营/名分/权力/资源变化、行动约束条件、关键物品流转、伏笔有关的信息。**

---

## 2. 项目边界

### 2.1 明确不做的事

本项目默认**不做**以下内容：

* 不把“智慧提炼”作为项目目标
* 不输出主题分析、寓意分析、价值判断
* 不写读后感
* 不做跨书混合处理
* 不在首次全书处理中做多章节并行
* 不引入 `paragraph_id`
* 不为复杂图谱而引入过重本体论
* 不自研 Agent 脚手架、LLM 调用层、工作流编排层

### 2.2 基本范围

* **每本书单独立项**
* 以原著已有章节划分为基础
* 只使用 `chapter_id`
* `chapter_id` 最多两级

  * 例如：`023`
  * 如内部必须暂时细分，可用 `023.1`、`023.2`
* 最终正式产物仍以章节为基本单位

### 2.3 执行边界

项目执行依赖宿主环境提供的多 Agent 能力与顺序式 runloop。

本地仅保留两类必要能力：

* 数据格式校验
* 由结构化数据到图和 Web 展示的生成

也就是说：

* 原著阅读、事实抽取、精简复述、状态更新、章节审计由多 Agent 协作完成
* 本地文件和代码只承担契约、落盘、校验和展示职责

---

## 3. 核心产物设计

### 3.1 第一优先级产物：精简版纯文本

这是项目的最重要交付物。要求是：

* 尽可能短
* 但不能丢失关键情节信息
* 保持连续叙述
* 保留必要的环境、人、物属性
* 读者只看这份文本，也能完整理解故事在这一章或整本书中发生了什么

它不是“摘要式提纲”，而是：

> **基于事实抽取后得到的高保真压缩叙述。**

---

### 3.2 第二优先级产物：JSON 真相层

这是系统内部的“真相源”。
精简版文本、时间线快照、人物关系图、故事发展图，都应由它派生。

JSON 真相层至少要覆盖：

* 本章关键事件链
* 人物属性变化
* 人物关系变化
* 阵营/名分/资源变化
* 关键物品流转
* 行动约束环境
* 伏笔与未决线索

---

### 3.3 第三优先级产物：画图数据层

至少包括两类：

#### A. 时间线快照图数据

目标是：

> 在故事推进到某个时间点时，能看到当下的人物关系、环境条件、关键物品、主要冲突等“截面”。

#### B. 简单直接的故事发展图数据

目标是：

> 用少量关键节点和清晰边类型，直接表达故事如何推进、哪里转折、哪里揭示、哪里冲突、哪里收束。

---

## 4. 全项目最重要的判断原则

这是整个系统的硬规则，必须固化进 Playbook、Prompt 和结构契约。

### 4.1 必须保留的内容

只要满足以下任一条件，就必须保留：

* 推动情节前进
* 改变人物状态
* 改变人物关系
* 改变阵营、名分、权力、资源
* 揭示动机、计划、误解、真相
* 对后文形成伏笔
* 构成行动约束的环境条件
* 关键物品、文书、法宝、兵器、信物的获得、转移或使用

### 4.2 可以压缩甚至省略的内容

满足以下条件时，可以压缩甚至省略：

* 纯装饰性景物描写
* 不影响情节的客套对话
* 重复性战斗铺陈
* 不改变人物、关系、情节的诗词铺陈
* 同义反复的情绪渲染

### 4.3 特别注意

**环境、人、物属性只有在“与情节因果相关”时才保留。**

也就是说：

* 环境不是一概保留，只保留影响行动、判断、冲突的环境
* 人物属性不是一概保留，只保留影响其行为、处境、关系、命运的属性
* 物品不是一概保留，只保留会推动情节或影响关系、权力、行动的物品

---

## 5. 核心执行方案

这里先讲执行方案，再讲多 Agent 协作。

## 5.1 总体思路

整个工程采用：

**单书立项 -> 逐章顺序阅读 -> 逐章抽事实 -> 逐章压缩复述 -> 逐章更新全书状态 -> 逐章审计 -> 全书收束**

注意，这里的核心不是“逐章总结”，而是：

> **逐章建立一个可继承的全书状态。**

因为后文理解依赖前文沉淀出来的：

* 人物注册表
* 别名与称谓归并
* 当前有效关系
* 当前阵营/名分/资源状态
* 关键物品归属
* 未决问题与伏笔
* 当前世界背景与局势

---

## 5.2 为什么首次处理不能多章节并行

这里把理由定为项目原则：

1. **人物识别依赖前文累积**
   很多人物在不同章节会以不同称呼、官职、绰号、法号、亲属称谓出现。
   不先顺序建立前文状态，后文很容易拆错实体。

2. **关系理解依赖前文历史**
   关系不是静态标签，而是逐步建立、强化、破裂、转化的。

3. **伏笔与误解依赖前文已知信息**
   有些内容在当章看像普通细节，但在前文状态背景下才知道它是伏笔或误导。

4. **压缩文本必须依赖已知背景浓缩表达**
   没有前文状态，模型往往会为了让本章单独可读而过度补背景，反而变长、变散。

所以，**首次生成必须顺序式处理**。
后续若要做补充审计、格式修整、局部回修，才可以有限并行。

---

## 5.3 单书执行流程

### 阶段 0：项目初始化

目标：建立最小但稳定的工程规则。

需要固定的内容：

* `book_id`
* 章节列表
* `chapter_id` 规则
* 事件 ID 规则
* 快照 ID 规则
* 关系类型表
* 物品流转类型表
* 环境约束表达规范
* 状态变化字段规范
* 输出文件清单

同时建立一个全书工作状态文件：

* `book_state.json`

它是后续每一章处理时，传递给下一章的最小工作记忆。

---

### 阶段 1：逐章处理循环

对每一章，严格按以下顺序执行。

#### Step 1：读取输入

输入只有两部分：

* 当前章节原文
* 已冻结的 `book_state.json`

不允许混入后文章节内容。
这样能避免 future leakage。

---

#### Step 2：抽取本章最小真相层

先不写压缩文本，先抽事实。

本章至少要抽出：

* 关键事件链
* 本章人物变化
* 本章关系变化
* 本章关键物品变化
* 本章环境约束变化
* 本章伏笔或未决线索
* 本章中需要继承到后文的背景信息

这里的关键原则是：

> **只抽与情节因果有关的事实，不抽无效装饰。**

---

#### Step 3：生成本章精简版纯文本

这一阶段才开始生成最重要的产物。

但要遵守一个铁规则：

> **精简版文本只能基于本章真相层生成，不能直接从原文自由发挥。**

这样做的目的是：

* 防止漏因果
* 防止漏状态变化
* 防止补写不存在的事实
* 防止模型被文风带偏，产出“好看但不准”的文本

---

#### Step 4：更新全书状态

本章通过后，将本章结果写入 `book_state.json`，至少更新以下部分：

* 人物注册表
* 别名表
* 当前有效关系
* 关系变化历史
* 关键物品归属与流转历史
* 当前环境与局势约束
* 未决问题 / 伏笔
* 已解决线索

这是下一章处理的基础。

---

#### Step 5：派生图数据

基于本章真相层和更新后的全书状态，生成：

* 本章关键时间点快照
* 本章故事发展图节点与边

注意：

* 图数据不是单独理解出来的
* 图数据必须从真相层派生
* 不允许“为了图好看”新增事实

---

#### Step 6：本章审计

本章必须经过忠实性审计。

审计重点：

* 有没有漏掉关键事件
* 有没有漏掉因果链
* 有没有漏掉关系变化
* 有没有漏掉关键物品流转
* 有没有漏掉环境约束
* 精简文本里是否出现真相层没有的事实
* 本章结果是否与前文状态冲突

未通过，就回修本章，不进入下一章。

---

#### Step 7：冻结本章结果并推进到下一章

只有本章通过后，才允许继续处理下一章。
这样全书状态才是稳定可继承的。

---

### 阶段 2：全书收束

全书逐章完成后，做全书级收束：

1. 拼接并轻量统一所有 `chapter_retelling.txt`
2. 输出全书精简版文本 `book_retelling.txt`
3. 输出人物注册表
4. 输出关系历史
5. 输出时间线快照集
6. 输出故事发展图
7. 输出未决问题清单或归并说明

注意：

**全书精简文本不应重新从整本原文再生成一遍。**
应当基于已审计通过的章节文本做轻量拼接与统一。
这样最稳。

---

## 6. 数据结构设计

这里采用“简单够用”的结构，不做过重设计。

## 6.1 本章真相层：`chapter_truth.json`

建议至少包含：

```json
{
  "book_id": "hongloumeng",
  "chapter_id": "023",
  "chapter_title": "示例标题",
  "events": [
    {
      "event_id": "023-E01",
      "order": 1,
      "summary": "某人物进入某场景，与另一人物发生关键接触。",
      "plot_role": "setup",
      "source_anchor": "章首：入场之后",
      "source_cues": ["人物A", "地点X", "初见"],
      "participants": ["entity_a", "entity_b"],
      "location": "location_x",
      "environment": ["夜间", "宴席环境，言行受礼法约束"],
      "important_objects": ["object_1"],
      "trigger": "前一局势推动其进入此场景",
      "result": "双方关系发生初步变化",
      "state_changes": [
        {
          "target": "entity_a",
          "field": "relation:entity_b",
          "change": "由疏远变为接触"
        }
      ],
      "certainty": "high"
    }
  ],
  "entity_changes": [
    {
      "entity_id": "entity_a",
      "display_name": "人物A",
      "aliases": ["别称A1"],
      "new_attributes": ["新出现的身份信息"],
      "changed_attributes": ["处境变化"]
    }
  ],
  "relation_changes": [
    {
      "source": "entity_a",
      "target": "entity_b",
      "relation_type": "alliance",
      "change": "new_or_changed",
      "reason_event_ids": ["023-E01"]
    }
  ],
  "object_changes": [
    {
      "object_id": "object_1",
      "change": "obtained",
      "holder": "entity_a",
      "reason_event_ids": ["023-E01"]
    }
  ],
  "environment_changes": [
    {
      "scope": "scene",
      "change": "night_fall",
      "effect": "行动受限，判断受环境影响"
    }
  ],
  "foreshadowing_or_open_threads": [
    "某人物的真实意图尚未完全揭示"
  ]
}
```

### 说明

这里不用 `paragraph_id`。
为了保留最小回查能力，保留两个轻量字段：

* `source_anchor`
* `source_cues`

这样既不增加太多复杂度，也不至于完全失去核对钩子。

---

## 6.2 全书工作状态：`book_state.json`

这是顺序处理的核心。

建议包含：

```json
{
  "book_id": "hongloumeng",
  "current_chapter_id": "023",
  "entity_registry": [],
  "relation_history": [],
  "object_history": [],
  "active_environment_constraints": [],
  "open_threads": [],
  "resolved_threads": [],
  "recent_context_digest": "供下一章使用的短摘要"
}
```

### 作用

它不是最终读者产物，而是：

> **下一章处理时的工作记忆与全书状态底座。**

---

## 6.3 时间线快照：`timeline_snapshots.json`

每个快照锚定一个关键事件之后的状态截面。

```json
[
  {
    "snapshot_id": "023-S01",
    "chapter_id": "023",
    "anchor_event_id": "023-E02",
    "order": 2,
    "location": "location_x",
    "characters_present": ["entity_a", "entity_b", "entity_c"],
    "active_relations": [
      {
        "source": "entity_a",
        "target": "entity_b",
        "relation_type": "alliance",
        "status": "fragile"
      }
    ],
    "environment_constraints": ["夜间", "礼法限制强"],
    "important_objects_in_play": ["object_1"],
    "current_conflicts": ["是否继续前进", "是否信任人物B"]
  }
]
```

### 设计原则

快照不必覆盖每个细节动作，只覆盖**关键时点**。
这样图才可读。

---

## 6.4 故事发展图：`story_flow_graph.json`

```json
{
  "nodes": [
    {
      "node_id": "023-T01",
      "chapter_id": "023",
      "label": "初次接触形成后续冲突条件",
      "based_on_event_ids": ["023-E01", "023-E02"]
    },
    {
      "node_id": "023-T02",
      "chapter_id": "023",
      "label": "误解发生，关系转紧张",
      "based_on_event_ids": ["023-E03"]
    }
  ],
  "edges": [
    {
      "from": "023-T01",
      "to": "023-T02",
      "type": "cause"
    }
  ]
}
```

### 边类型建议固定为少数几类

* `next`
* `cause`
* `reveal`
* `conflict`
* `resolve`

这样结构清楚，不会画成杂乱的蜘蛛网。

---

## 7. 质量标准与验收标准

一个章节只有同时满足以下条件，才算通过：

1. **关键情节转折完整**
   没漏掉推动本章发展的关键事件。

2. **因果链完整**
   读精简文本时，能看出“为什么发生、如何变化、导致什么结果”。

3. **状态变化完整**
   人物状态、关系变化、阵营/名分/资源变化都有记录。

4. **关键物品完整**
   关键物品的获得、转移、使用没有漏。

5. **环境约束完整**
   影响行动的环境条件没有被删掉。

6. **精简文本不越界**
   精简文本中没有真相层不存在的事实。

7. **前后章状态连续**
   本章结果与前文沉淀状态不冲突。

8. **伏笔与未决问题可继承**
   需要传递到后文的信息已进入全书状态。

## 8. 多 Agent 协作方案

这里按“先执行方案，再协作方案”的顺序收束。

本项目的多 Agent 方案，不以“多章节并行”为核心，而以：

> **顺序式逐章处理中的角色分工与相互制衡** 为核心。

## 8.1 推荐团队结构

建议采用 **1 个主编排 + 4 个执行角色** 的结构。

### 1. Coordinator（主编排）

职责：

* 维护项目流程
* 控制章节推进顺序
* 传递 `book_state.json`
* 冻结章节结果
* 发起回修
* 负责最终产物清单

行为要求：

* 只负责调度与状态推进
* 不擅自改写事实
* 不跳过审计
* 不允许后文信息提前进入前文章节

---

### 2. Extractor（事实抽取 Agent）

职责：

* 阅读当前章节
* 生成 `chapter_truth.json` 初稿
* 识别事件、状态变化、关系变化、环境约束、物品流转、伏笔

行为要求：

* 只抽事实，不做评论
* 不做智慧提炼
* 不做主题分析
* 不确定就标记，不脑补
* 环境、人、物属性只保留因果相关项

---

### 3. Compressor（精简复述 Agent）

职责：

* 基于 `chapter_truth.json` 生成 `chapter_retelling.txt`

行为要求：

* 只能依据 truth 写
* 不能补充新事实
* 目标是“尽可能短，但不丢关键因果信息”
* 不写评论
* 不写主题
* 不写提纲式总结
* 只输出连续叙述文本

---

### 4. State Keeper（状态与图数据 Agent）

职责：

* 更新 `book_state.json`
* 归并人物与别名
* 更新关系历史
* 更新关键物品流转
* 生成快照与故事图数据

行为要求：

* 不确定时不强行归并
* 图数据必须从 truth 派生
* 不发明节点，不发明关系
* 保持全书状态连续稳定

---

### 5. Auditor（忠实性审计 Agent）

职责：

* 检查章节输出质量
* 检查 truth 与 retelling 是否一致
* 检查是否漏项、错项、幻觉
* 检查与前文状态是否冲突

行为要求：

* 专门找错，不负责美化
* 默认怀疑遗漏和补写
* 发现问题先给修复意见，再回投责任 Agent
* 不直接重写整个章节

---

## 8.2 团队运行方式

### 每章标准流程

1. Coordinator 取出当前章节与已冻结 `book_state`
2. Extractor 产出 `chapter_truth.json`
3. Compressor 基于 truth 产出 `chapter_retelling.txt`
4. State Keeper 更新 `book_state`，生成本章图数据
5. Auditor 审计
6. 若失败，回投给对应 Agent 修复
7. 通过后冻结本章，进入下一章

### 注意

* **章节之间不并行**
* **角色可以协作，但章节必须顺序推进**
* **首次全书跑通后，才考虑局部审计或格式修整的辅助并行**

## 9. Team Playbook

## Playbook 1：项目初始化

目标：建立单书工程规则。

内容包括：

* 书名、`book_id`
* 章节列表

* truth schema

## Playbook 2：单章处理

目标：以当前章节为单位，完成本章全部产物。

顺序固定为：

1. 读取当前章节与前文状态
2. 抽取真相层
3. 生成精简文本
4. 更新全书状态
5. 生成图数据
6. 审计
7. 冻结

## Playbook 3：回修

触发条件包括：

* 漏掉关键事件
* 漏掉关系变化
* 漏掉物品流转
* 漏掉环境约束
* 精简文本有新增事实

* 人物归并错误

修复原则：

* 只修具体问题
* 不整章推倒重来，除非错误已经污染整章结果
* 修完后必须重新审计

## 10. 可直接给 Agent Team 的总控 Prompt

下面这段可以直接作为项目的总控 Prompt 基础版。

```text
你是“名著事实提取项目”的主编排 Agent。

【项目定位】
本项目不是智慧提炼项目，不做主题分析，不做读后感，不做价值判断。
本项目只做三类产物：
1. 不丢失关键情节因果信息的精简纯文本（第一优先级）
2. 机器可理解的 JSON 真相层（第二优先级）
3. 可直接画图的数据层（第三优先级）

【忠实的定义】
不得丢失以下信息：
- 推动情节前进
- 改变人物状态
- 改变人物关系
- 改变阵营、名分、权力、资源
- 揭示动机、计划、误解、真相
- 对后文形成伏笔
- 构成行动约束的环境条件
- 关键物品、文书、法宝、兵器、信物的获得、转移或使用

允许压缩甚至省略的内容：
- 纯装饰性景物描写
- 不影响情节的客套对话
- 重复性战斗铺陈
- 不改变人物、关系、情节的诗词铺陈
- 同义反复的情绪渲染

特别注意：
- 环境、人、物属性只有在与情节因果相关时才保留

【工程边界】
- 每本书单独立项
- 以章节为基本处理单位
- 只使用 chapter_id，不使用 paragraph_id
- chapter_id 最多两级
- 首次全书处理必须逐章顺序推进，不做多章节并行
- 不允许使用后文知识反写前文章节
- 不确定的信息必须标记，而不是脑补抹平

【执行顺序】
虽然精简纯文本是第一优先级产物，但执行顺序必须是：
原文 -> 最小真相抽取 -> 精简文本 -> 状态更新 -> 图数据派生

【章节级必须产出】
1. chapter_truth.json
2. chapter_retelling.txt
3. timeline_snapshots.json 中的本章快照部分
4. story_flow_graph.json 中的本章节点与边
5. 更新后的 book_state.json

【chapter_truth.json 至少包含】
- book_id
- chapter_id
- chapter_title
- events[]
- entity_changes[]
- relation_changes[]
- object_changes[]
- environment_changes[]
- foreshadowing_or_open_threads[]

【events[] 每个事件至少包含】
- event_id
- order
- summary
- plot_role
- source_anchor
- source_cues
- participants
- location
- environment
- important_objects
- trigger
- result
- state_changes
- certainty

【团队角色】
1. Coordinator：调度流程与推进章节，不改写事实
2. Extractor：抽取当前章节真相层
3. Compressor：基于真相层写精简文本
4. State Keeper：更新全书状态并派生图数据
5. Auditor：检查漏项、错项、幻觉与前后冲突

【章节推进规则】
- 只有当前章节审计通过，才允许进入下一章
- 若发现问题，必须回修本章
- book_state 只能由已冻结章节结果更新

【验收标准】
一个章节只有同时满足以下条件才算通过：
1. 关键情节转折完整
2. 因果链完整
3. 人物状态和关系变化完整
4. 关键物品流转完整
5. 环境约束完整
6. 精简文本没有新增 truth 中不存在的事实
7. 与前文状态连续一致
8. 需要传递到后文的伏笔和未决问题已进入 book_state

开始执行时，先建立单书项目，再按章节顺序处理。
```

## 11. 各角色行为 Prompt

### Extractor

```text
你是 Extractor。你的唯一任务是从当前章节抽取与情节因果有关的事实，输出 chapter_truth.json 初稿。

你必须保留：
- 推动情节前进的内容
- 人物状态变化
- 人物关系变化
- 阵营、名分、权力、资源变化
- 动机、计划、误解、真相
- 伏笔
- 行动约束环境
- 关键物品流转

你可以压缩或忽略：
- 纯装饰性景物
- 不影响情节的客套对话
- 重复性铺陈
- 不承担情节功能的诗词
- 同义反复的情绪描写

禁止：
- 主题分析
- 智慧提炼
- 价值判断
- 使用后文知识
- 脑补不确定信息
```

### Compressor

```text
你是 Compressor。你的唯一任务是基于 chapter_truth.json 生成 chapter_retelling.txt。

要求：
- 只能依据 truth 写
- 不得新增 truth 中不存在的事实
- 文本尽量短
- 但不得丢失关键情节因果信息
- 保留与情节因果相关的环境、人、物属性
- 只写连续叙述文本，不写提纲，不写评论
```

### State Keeper

```text
你是 State Keeper。你的任务是把当前章节结果并入全书状态，并派生图数据。

你必须维护：
- entity_registry
- relation_history
- object_history
- active_environment_constraints
- open_threads
- timeline_snapshots
- story_flow_graph

规则：
- 不确定时不要强行合并人物
```

### Auditor

```text

你必须检查：

- 是否漏掉因果链
- 是否漏掉关系变化
- 是否漏掉关键物品流转
- 是否漏掉环境约束
- 是否存在幻觉补写

输出：
- 问题类型
- 涉及章节
- 涉及对象

```

## 12. 最终收束

这个项目不是“让 LLM 写摘要”，而是“让 LLM 在逐章顺序阅读中，先建立最小真相层，再生成不丢失关键情节信息的精简文本，并同步维护全书状态与画图数据”。

因此，整个方案的正确重心是：

* **目标重心**：精简版纯文本
* **工程重心**：真相层与全书状态
* **流程重心**：逐章顺序推进
* **协作重心**：角色分工，而不是多章节并行
* **质量重心**：忠实性审计

这就是目前最稳、最适合交给 Agent Team 执行的方案。
